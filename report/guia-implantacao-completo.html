1Contexto e objetivo

A Boa Vista solicitou visibilidade em tempo real do consumo do GitHub Copilot pelos desenvolvedores, cobrindo quatro pilares:

  1. FinOps — tokens consumidos e custo estimado, com breakdown por usuário, time e sessão
  2. Performance (SRE) — latência das requisições aos modelos e identificação de gargalos
  3. Erros e limites — visibilidade sobre rate limits (HTTP 429) e falhas de comunicação
  4. Adoção — quais modelos estão sendo mais usados e por quais desenvolvedores

Esta solução implanta um pipeline OpenTelemetry nativo que captura, em tempo real, todas as interações dos devs com o Copilot Chat no VS Code, envia para o Dynatrace, e disponibiliza os dados na app nativa AI & LLM Observability — sem que nenhum dev precise fazer configuração manual.

Por que agora é possível A Microsoft/GitHub lançaram em julho de 2026 o suporte OpenTelemetry nativo para o Copilot Chat no VS Code (Monitor agent usage with OpenTelemetry) e o Enterprise managed-settings.json em GA, permitindo enforcement centralizado. Isso viabiliza pela primeira vez a captura em tempo real do consumo do Copilot na IDE, com granularidade de span por chamada.

Fluxo de dados

  1. Dev abre o VS Code — extensão Copilot Chat detecta github.copilot.chat.otel.enabled=true via managed policy
  2. Dev faz uma pergunta ao Copilot Chat — extensão gera árvore de spans (invoke_agent → N × chat → N × execute_tool)
  3. Spans são enviados via OTLP HTTP para o OTel Collector interno na rede corporativa
  4. Collector adiciona o header Authorization: Api-Token dt0c01.xxx e forwarda via HTTPS para o Dynatrace
  5. Grail armazena; app AI & LLM Observability detecta automaticamente o serviço copilot-chat
  6. Dashboards, notebook e DQL passam a consultar os dados

5Passo 1 — Configurar o Dynatrace

Responsável: Admin do tenant Dynatrace  ·  Tempo estimado: 15 minutos

5.1 Criar API Token classic para OTLP ingest

  1. Acesse o tenant: https://fov31014.apps.dynatrace.com
  2. Ctrl+K → digite Access Tokens → Generate new token
  3. Nome sugerido: copilot-otlp-ingest
  4. Selecionar os scopes:
    • openTelemetryTrace.ingest — para receber spans
    • metrics.ingest — para receber metrics
  5. Generate — o token começa com dt0c01. (~96 caracteres)
  6. Salvar no cofre corporativo (1Password, HashiCorp Vault, AWS Secrets Manager, etc.)
Segurança Este token nunca é distribuído para máquinas de devs — fica apenas no OTel Collector. Isso permite rotação sem redeploy nos devs e reduz drasticamente a superfície de exposição.

5.2 (Opcional) Preparar dashboard e notebook

Os assets prontos desta POC estão em:

  • dashboards/copilot-chat-observability.json — importar via Ctrl+K → Dashboards → + Dashboard → Upload
  • Notebook v2 já criado no tenant: abrir aqui

6Passo 2 — Subir o OTel Collector

Responsável: Plataforma / SRE da Boa Vista  ·  Tempo estimado: 1 dia útil (inclui provisionamento e liberação de firewall)

6.1 Arquivo compose.yml

services:
  otel-collector:
    image: otel/opentelemetry-collector-contrib:latest
    container_name: bv-otel-collector
    restart: unless-stopped
    command: ["--config=/etc/otel-collector-config.yaml"]
    volumes:
      - ./otel-collector-config.yaml:/etc/otel-collector-config.yaml:ro
    env_file: .env
    ports:
      - "4318:4318"     # OTLP HTTP (VS Code envia aqui)
      - "4317:4317"     # OTLP gRPC (alternativa)
      - "13133:13133"   # Health check
    healthcheck:
      test: ["CMD", "wget", "-qO-", "http://localhost:13133"]
      interval: 30s

6.2 Arquivo otel-collector-config.yaml

receivers:
  otlp:
    protocols:
      http:
        endpoint: 0.0.0.0:4318
        max_request_body_size: 4194304
      grpc:
        endpoint: 0.0.0.0:4317

processors:
  batch:
    timeout: 5s
    send_batch_size: 512
  resource:
    attributes:
      - key: deployment.environment
        value: production
        action: upsert

exporters:
  otlphttp/dynatrace:
    endpoint: ${env:DT_OTLP_ENDPOINT}
    headers:
      Authorization: "Api-Token ${env:DT_INGEST_TOKEN}"
    compression: gzip

extensions:
  health_check:
    endpoint: 0.0.0.0:13133

service:
  extensions: [health_check]
  pipelines:
    traces:
      receivers: [otlp]
      processors: [batch, resource]
      exporters: [otlphttp/dynatrace]
    metrics:
      receivers: [otlp]
      processors: [batch, resource]
      exporters: [otlphttp/dynatrace]

7Passo 3 — Configurar o GitHub Enterprise

Responsável: Admin do GitHub Copilot Enterprise da Boa Vista  ·  Tempo estimado: 1 dia útil

7.1 Selecionar a organização hospedeira

  1. Login no GitHub Enterprise Settings
  2. Ir em AI Controls → Agents
  3. Selecionar a organização que hospedará o .github-private (geralmente a org corporativa central)

7.2 Criar o repositório .github-private

Se ainda não existir, criar um repositório privado com esse nome exato na organização selecionada. É onde ficam configurações que se aplicam a toda a org sem serem visíveis publicamente.

7.3 Publicar o arquivo copilot/managed-settings.json

Path exato no repositório: copilot/managed-settings.json, no branch default (main).

{
  "telemetry": {
    "enabled": true,
    "endpoint": "https://otel-collector.boavista.corp:4318",
    "protocol": "otlp-http",

    "captureContent": false,
    "lockCaptureContent": true,

    "serviceName": "copilot-chat-boavista",
    "resourceAttributes": {
      "deployment.environment": "production",
      "organization": "boavista",

Tempo de propagação: até 1h O VS Code faz pull das managed settings a cada 1h e ao fazer login. Depois de commitar o arquivo, os devs recebem a configuração sem fazer nada. Para forçar antes disso, o dev pode simplesmente reiniciar o VS Code.

8Passo 4 — Piloto no VS Code (opcional, para 3-5 devs voluntários antes do rollout enterprise)

Responsável: Dev voluntário  ·  Tempo estimado: 5 minutos por dev

Por que fazer piloto? Enquanto o time de GitHub Admin prepara e publica o managed-settings.json, você pode validar a arquitetura com 3-5 devs configurando manualmente no user settings deles. Quando o rollout enterprise acontecer, o managed-settings sobrescreve automaticamente sem que os devs precisem desfazer nada.

8.1 Configurar o user settings do VS Code

  1. Cmd/Ctrl+Shift+P → digite Preferences: Open User Settings (JSON)
  2. Adicionar as 5 linhas de OTel dentro das chaves {}:
{
  // ...suas outras settings...

  "github.copilot.chat.otel.enabled": true,
  "github.copilot.chat.otel.exporterType": "otlp-http",
  "github.copilot.chat.otel.otlpEndpoint": "http://otel-collector.boavista.corp:4318",
  "github.copilot.chat.otel.captureContent": false,
  "github.copilot.chat.otel.maxAttributeSizeChars": 0
}

8.2 Reload do VS Code

Cmd/Ctrl+Shift+P → Developer: Reload Window

8.3 (Opcional) Enriquecer com identidade do dev

Para popular user.email, user.team, cost_center em cada span, adicionar variável de ambiente no shell:

Fish shell
set -Ux OTEL_RESOURCE_ATTRIBUTES \
  "user.email=nome.sobrenome@boavista.com.br,user.name=Nome Sobrenome,user.team=plataforma-api,cost_center=engineering"
Bash / Zsh
echo 'export OTEL_RESOURCE_ATTRIBUTES="user.email=nome.sobrenome@boavista.com.br,user.team=plataforma-api"' >> ~/.zshrc
Herança da variável no VS Code No macOS, se o dev abrir o VS Code pelo ícone do dock (Finder), a variável de ambiente do shell NÃO é herdada. Precisa fechar completamente (Cmd+Q) e reabrir pelo terminal com code /caminho/do/workspace. Windows/Linux herdam normalmente.

8.4 No rollout enterprise (via managed-settings.json)

Nada. Os devs não fazem absolutamente nada. Ao próximo login/reinício do VS Code, a policy é aplicada automaticamente e sobrescreve qualquer user setting anterior. Se o time de Plataforma distribuir também a variável OTEL_RESOURCE_ATTRIBUTES via Intune/Jamf/GPO, nem mesmo isso os devs precisam configurar.

11Considerações de LGPD

11.1 Modo padrão — captureContent = false

LGPD-safe Apenas metadados são capturados: modelo usado, contagens de token, latência, nomes de ferramentas invocadas, IDs (session, conversation, trace, span), metadados git (branch, commit, repo).

Não é capturado: texto de prompts, respostas do modelo, código-fonte, conteúdo de arquivos abertos, comandos de terminal.

Risco de PII: zero. Não requer aprovação do DPO.

11.2 Modo com conteúdo — captureContent = true

Trilha formal de aprovação necessária Ativar captureContent=true captura texto integral de prompts, respostas, argumentos de ferramentas (inclui caminhos de arquivos e código-fonte lido) e comandos de terminal.

Antes de habilitar em produção:
  • Aprovação do DPO da Boa Vista
  • Revisão pelo jurídico trabalhista
  • Comunicação transparente aos devs (aviso interno)
  • Habilitar redação de PII no OTel Collector (CPF, email, tokens, senhas)
  • Definir política de retenção no Grail (default 35d)

11.3 Recomendação

Iniciar a POC e o rollout enterprise com captureContent=false. Cobre 100% dos requisitos formais de FinOps + SRE. A opção de captura de conteúdo pode ser habilitada depois, em subset de devs, se surgir demanda específica de análise de qualidade de prompts, após completar a trilha de aprovação.

12.2 Troubleshooting comum

VS Code não emite nada

  1. Verificar policy: Cmd/Ctrl+Shift+P → Developer: Policy Diagnostics — deve mostrar telemetry.enabled = true
  2. Trocar temporariamente exporterType para "console" e ver spans no Output panel
  3. Verificar conectividade: curl -v http://otel-collector.boavista.corp:4318/v1/traces -H "Content-Type: application/json" -d "{}"
  4. Aumentar log level: setar env COPILOT_OTEL_LOG_LEVEL=debug

Collector recebe, Dynatrace não mostra

  • Ver logs: docker logs -f bv-otel-collector — procurar HTTP 401 (token inválido), 403 (scope faltando)
  • Verificar endpoint: deve ser *.live.dynatrace.com, não *.apps.dynatrace.com
  • Confirmar scopes do token: openTelemetryTrace.ingest + metrics.ingest

Atributos user.email/user.team vazios

OTEL_RESOURCE_ATTRIBUTES não foi configurado ou o VS Code não herdou a variável do shell (ver Seção 8.3). Alternativa mais robusta: adicionar processor de enrichment no OTel Collector que faz lookup do email do dev contra um CSV/AD/LDAP.
